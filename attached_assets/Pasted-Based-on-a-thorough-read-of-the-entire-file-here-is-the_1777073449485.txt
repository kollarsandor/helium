15. `RateLimiter.acquire`: The `while (true)` loop has `defer self.mutex.unlock()` inside it — `defer` runs at the end of the block scope; because `self.mutex.lock()` and `defer self.mutex.unlock()` are inside `while (true)`, the mutex is unlocked via `defer` at the end of each `while` body iteration, but then `self.mutex.unlock()` is also called explicitly before `std.time.sleep(...)` and `self.mutex.lock()` is called again after, resulting in a double-unlock when the deferred unlock fires — undefined behavior / panic.
16. `RateLimiter.acquire`: `self.mutex.lock()` is called again after the explicit `self.mutex.unlock()` before sleep, but `defer self.mutex.unlock()` was already registered for this iteration — when the iteration ends after re-locking, `defer` fires a second unlock on an already-unlocked mutex.
17. `RateLimiter.acquire`: `const wait_ms = @max(0, wait_until - now_ms)` — `wait_until` and `now_ms` are `i64`; if `wait_until - now_ms` is negative, `@max(0, ...)` is fine, but the subsequent `if (wait_ms <= 0) continue;` check is unreachable because `@max(0, x)` is always `>= 0`, yet the code still calls `continue` for `wait_ms == 0`, causing a wasted iteration instead of sleeping.

20. `saveSessionsToDisk`: `sessions_mutex.lock()` is called manually without `defer sessions_mutex.unlock()` — the unlock happens manually later, but if any `catch` between lock and unlock fires and returns early, the mutex is left permanently locked (deadlock).

27. `callExaSingle`: The parsed JSON result `parsed.value` is returned directly after `parsed.deinit()` is deferred — `parsed.deinit()` will free the memory backing `parsed.value` at the end of scope, but the caller receives a dangling/freed `std.json.Value`. This is a use-after-free bug.
28. `callExaSingle`: The `auth_header` is formatted as just `"{s}"` with the raw API key — the Exa API uses the header `x-api-key`, which is added separately in the `headers` array, but `auth_header` is allocated and immediately used only as the `x-api-key` value, so the extra `allocPrint` formatting is wasteful and misleading (should just use the key directly).
29. `callExaSingle`: Inside the retry loop, `var client = std.http.Client{ .allocator = allocator }; defer client.deinit();` is declared inside the `while` loop body — because `defer` is scoped to the block, each loop iteration creates and destroys a new HTTP client, which is correct but causes repeated TLS/connection setup overhead; however, the real bug is that if `continue` is called after a failed attempt, the `defer client.deinit()` still fires, which is correct, but the `response_body` ArrayList declared with `defer response_body.deinit()` inside the loop will also be deinitialized on each `continue`, causing its data to be freed — this is actually correct per Zig semantics, but the re-declaration of `var response_body` in the same loop without a new `var` means it is re-initialized each iteration, which is fine.
30. `callExaSingle`: `return error.ExaFailed` at the end of the function is unreachable — the `while (attempt < 3)` loop always exhausts 3 iterations before reaching it, and all paths inside either `return` or `continue`, so this is dead code, but more critically the function will actually reach this line after 3 failed retries only if no `return` was hit, which could happen if `result.status` is not `.too_many_requests` and not `>= 500` and the JSON parse succeeds but `parsed.value` is returned — but that return is inside the loop. If the loop ends without returning, `ExaFailed` is returned — this is technically correct logic but the `callExaSingle` return at the bottom is unreachable when the loop returns the parsed value. Actually the bug is that `return parsed.value` inside the loop returns from the function with a freed value (see #27), so control never reaches `return error.ExaFailed` in the success case.
31. `ExaMultiResult.deinit`: `self.results` (of type `?std.json.Value`) is never freed/deinitialized in `deinit` — if `results` contains a parsed JSON value, its memory is leaked.
32. `ExaSearchTask`: The `thread` field is of type `?std.Thread`, but `ExaSearchTask` is never actually used with threads anywhere in the code — `executeExaToolCall` calls `callExaSingle` sequentially in a loop instead of spawning threads, making `ExaSearchTask` and its `run` method dead/unused code. 33. `normalizeExaQueries`: The `seen` map's keys are allocated with `allocator.dupe(u8, lower)` and stored in the map, but when the function returns, `defer seen.deinit()` is called — however `StringHashMap.deinit()` does not free the keys, so all the `lower_key` strings are leaked.
34. `formatExaResultsForModel`: `var truncated = false` is declared and set but never used in the return value or any output — it tracks whether truncation occurred but the information is discarded.
35. `formatExaResultsForSse`: When `res_array` is not found (`orelse` branch), the code checks `entry.err` and appends an error item, then calls `continue` — but if `entry.err` is null, the `continue` is still executed (after the `if (entry.err)` block does nothing), silently skipping the entry, which is correct behavior but the logic structure is confusing and the `orelse { ... continue; }` block will `continue` regardless of whether an error item was appended.
36. `SseWriter.writeComment`: This function is identical to `SseWriter.ping` — both write `": ping\n\n"` — one of them is redundant/misnamed (a comment in SSE should be `": comment text\n\n"`, not a ping).
37. `writeJsonEvent`: For pointer types, only `[]u8` (i.e., `ptr.child == u8`) is handled; all other pointer types fall through to `writer.writeAll("null")` — this silently serializes complex pointer values as JSON `null` without any error or warning, which is a silent data loss bug.
38. `writeJsonEvent`: The `@typeInfo(ValType)` match on `.pointer` checks `ptr.child == u8` but does not check that it is a slice (`ptr.size == .Slice`) — a single-item pointer to `u8` would also match and be incorrectly passed to `writeJsonString` as a slice, causing undefined behavior.
39. `parseSseLine`: `parsed.deinit()` is deferred, but string fields copied via `allocator.dupe(u8, ...)` from `parsed.value` reference memory owned by `parsed` — the duplication is correct, but after `deinit()` fires the original strings in `parsed` are freed while the chunk holds its own copies, so this is actually correct. However, `chunk.tool_calls` elements' `arguments` strings are duped from `a.string`, which is owned by `parsed`. Since `parsed` is deferred-deinitialized at function return and the duplication happens before return, this is safe. No bug here — this note cancels itself.
40. `ToolCallAccumulator.getArguments`: If `total == 0` (no parts), `allocator.alloc(u8, 0)` returns an empty slice — this is valid in Zig but the caller receives an empty string, which may be mishandled downstream as no arguments.

49. `callOpenAIStreaming`: After reading stdout to EOF and before reading stderr, if stdout reading sets `read_err`, the function still proceeds to read stderr — but `child.stderr` reading with `readAllArrayList` may block if the child process is still running. Since stdout was already closed (EOF), the child should be done, but there's no guarantee, creating a potential hang.
50. `callOpenAIStreaming`: `child.stdin` is closed by calling `stdin.close()` and setting `child.stdin = null` — however, if `child.spawn()` fails, `stdin` was never opened, and the `if (child.stdin) |stdin|` guard handles that. But if `stdin.writeAll(body)` fails (the `catch {}` silently swallows the error), the child process may never receive input and will hang waiting for stdin, causing `child.wait()` to block indefinitely.
51. `callOpenAIStreaming`: The curl `--fail-with-body` flag causes curl to exit with code 22 on HTTP errors while still outputting the body — but the code treats any non-zero curl exit code as `error.CurlFailed` without attempting to parse the error body, so HTTP error responses from the AI API are silently discarded with a generic error.
52. `chatStreamCallback`: Two separate `ev_buf` variables are declared in the same scope (one for `reasoning_start` and one for the reasoning text event) — in Zig, variables cannot be redeclared in the same block scope; this is a compile error (shadowing).
53. `chatStreamCallback`: Same double-declaration issue with `ev_buf` in the `c.content` branch — two `var ev_buf` declarations in the same `if (c.content)` block.
54. `getSystemPrompt`: `total_minutes` is computed as `@divTrunc(@mod(now_ms, 86400_000), 60000)` — `now_ms` is in milliseconds, `@mod(now_ms, 86400_000)` gives milliseconds within the current day, and dividing by 60000 gives minutes. This is correct. However, `hour` and `minute` are then derived from `total_minutes` correctly. No bug here — this note cancels itself.
55. `getSystemPrompt`: The time is presented as UTC but `std.time.milliTimestamp()` returns wall-clock time which may or may not be UTC depending on system settings — on most servers it is UTC, but there is no explicit UTC conversion, making this potentially wrong in non-UTC environments.
56. `pruneDeletedSessionsUnlocked`: Called without holding `deleted_sessions_mutex` (it is called from within functions that hold it, which is correct), but the function name says "Unlocked" suggesting it should be called while the lock is held — however `isDeleted` calls `pruneDeletedSessionsUnlocked` after locking the mutex, and `markDeleted` does too. This is correct, but `pruneDeletedSessionsUnlocked` also calls `nowSeconds()` twice (once in `isDeleted` before the call and once implicitly — actually it takes `now` as a parameter, so only one call). No bug.
57. `isDeleted`: Calls `pruneDeletedSessionsUnlocked(nowSeconds())` and then immediately calls `nowSeconds()` again for the TTL check — two separate clock reads may give slightly different values, but this is a minor inconsistency, not a real bug.
58. `evictSessions`: `session_locks_guard.lock()` with `defer session_locks_guard.unlock()` is acquired first, then `sessions_mutex.lock()` with `defer sessions_mutex.unlock()` — lock ordering is consistent here. However, other parts of the code acquire `sessions_mutex` first and then may try to acquire `session_locks_guard` — if lock ordering is not consistent across the codebase, a deadlock can occur.
59. `evictSessions`: After evicting sessions from `sessions_map`, the corresponding entries in `session_locks_map` and `session_exa_limiters_map` are never removed — leaked mutex and limiter objects accumulate indefinitely for evicted sessions.
60. `evictSessions`: After evicting sessions from `sessions_map`, the evicted session IDs are not added to `deleted_sessions_map` — this means clients can re-create an evicted session (due to TTL) as a fresh session on the next request, which may be intentional, but eviction due to `MAX_SESSIONS` overflow is silent and the client has no way to know.
61. `stripOrphanedAssistantToolCalls`: `tool_ids_present` stores raw `[]const u8` keys from message `tool_call_id` fields without duplication — these point into the `messages` list which is being simultaneously iterated and then replaced. If the messages list is freed and recreated, these raw pointers become dangling. The keys are used before the messages are freed, so this is safe in practice, but fragile.
62. `stripOrphanedAssistantToolCalls`: When an assistant message with tool calls has all its tool calls stripped (none are present in `tool_ids_present`) and its content text is empty, the message is silently dropped entirely — this can cause the conversation history to lose an assistant turn without any indication.
63. `trimHistory`: `var cut2 = findTurnBoundary(history.items, 1)` — the `if (cut2 <= 0 ...)` check treats `cut2 == 0` as needing adjustment, but `cut2` is `usize` which is always `>= 0`, so `cut2 <= 0` only catches `cut2 == 0`. This is technically correct but `<= 0` on a `usize` is misleading and should be `== 0`.
64. `buildApiMessages`: Duplicate call to `stripOrphanedAssistantToolCalls` — called once after initial clone, once after trimming by count, and once inside the size-trimming loop. While not incorrect, this is redundant and wasteful.
65. `findTurnBoundary`: If all messages from `min_idx` onwards have `role != "user"` and the fallback also finds no boundary, the function returns `messages.len`, which when used as a cut index would delete all messages — this is a logical error in edge cases where no user message exists in the window.
66. `findToolBoundary`: The loop increments `idx` when a "tool" role message is found at the start but this is incorrect — "tool" messages at the beginning of the window should not be skipped alone; the function is supposed to find where to safely cut without splitting a tool-call chain, but orphaned leading "tool" messages are not properly handled.
67. `messageCharSize`: Takes `msg: *Message` (mutable pointer) only to set `cached_size` — but callers pass `@constCast(msg)` to convert from `const Message` to `*Message`, which is undefined behavior if the original was declared `const`.
68. `safeMaxTokens`: Calls `messageCharSize(@constCast(msg))` — same undefined behavior as #67 for `const` messages.
69. `safeMaxTokens`: The token estimation `total_chars / 3` is a very rough heuristic (actual token:character ratio varies by language and model) — for non-ASCII (e.g., Chinese/Japanese/Korean) content, actual token counts may be much higher, leading to `max_tokens` being set too high and causing API errors.
70. `generateUuid`: The UUID format uses lowercase hex `{x:0>2}` — standard UUIDs use lowercase hex, which is correct, but the format string produces the bytes without any leading-zero guarantee for the format specifier combination. Actually `{x:0>2}` does guarantee 2-digit output with leading zeros — no bug.
71. `generateUuid`: UUID v4 sets bits correctly (`bytes[6] = (bytes[6] & 0x0f) | 0x40` for version, `bytes[8] = (bytes[8] & 0x3f) | 0x80` for variant) — this is correct.
72. `safeB64Decode`: `const pad = 4 - (stripped.len % 4)` — when `stripped.len % 4 == 0`, `pad == 4`, and the condition `if (pad != 4)` skips padding, which is correct. However, `stripped.len % 4 == 0` could mean either a perfectly padded string or a valid unpadded Base64 string — for URL-safe Base64 without padding, this case is handled correctly. No bug.
73. `safeB64Decode`: The `is_url_safe` detection `std.mem.indexOfAny(u8, padded, "-_") != null` fires on the padded string which may now contain `=` padding that was just added — but `=` is not `-` or `_`, so this is fine. However, if the original string has both standard (`+`, `/`) and URL-safe (`-`, `_`) characters mixed, only URL-safe conversion is applied, corrupting the standard characters. This is an edge case but a real bug.
74. `decodeAndStripB64`: The `stripped` string returned to caller is the Base64 string after stripping whitespace and removing data URL prefix — this is used later as the `data` field stored in the message. But `stripped` is allocated separately from `decoded` and both must be freed by the caller — the `normalizeInputImage` function correctly frees them on error paths but returns `.data = result.stripped` on success, meaning the caller owns `stripped`. This is correct but the ownership transfer is undocumented and error-prone.
75. `normalizeInputImage`: Returns `.media_type = try allocator.dupe(u8, detected.mime)` where `detected.mime` is a string literal — the duplication is unnecessary (string literals have static lifetime) but harmless.
76. `pathInProject`: `std.mem.startsWith(u8, full_path, project_root)` does not check that `project_root` ends with a path separator — a `project_root` of `/foo` would incorrectly allow `/foobar/baz` to pass as "in project". This is a path traversal logic error.
77. `pathInProject`: If `project_root` is empty (its initial value is `&[_]u8{}`), then `std.mem.startsWith(u8, full_path, "")` is always true, meaning every path passes the check — complete path traversal vulnerability if `project_root` is not set.
78. `executeAgentTool` (`read_file`): `total_lines += 0` in the `else` branch (when `line_num < offset`) is a no-op — it should probably be `total_lines += 1` to count lines skipped before the offset window. The total line count will be undercounted by the number of pre-offset lines.
79. `executeAgentTool` (`write_file`): `std.fs.makeDirAbsolute(parent)` is called to create the parent directory, but it only creates one level — if the parent path has multiple non-existent levels, this will fail silently (the error is ignored with `catch {}`), and the subsequent `createFileAbsolute` will also fail, returning an error to the agent.

81. `executeAgentTool` (`list_directory`): Hidden files (starting with `.`) are filtered out except `.replit` and `.env` — this hardcoded allowlist will miss other important dotfiles (e.g., `.gitignore`, `.zig-cache`) and silently omit them from directory listings.
82. `handleChatEndpoint`: The model is hardcoded as `"zai-org/glm-5.1"` — but `extra_body_thinking: true` is passed, which enables chain-of-thought for this model. If the model is changed without updating the `extra_body_thinking` flag, the API call may fail or produce unexpected output.
83. `handleAgentEndpoint`: Uses `AgentStreamContext` with field `.sse_writer = &conn` — but `AgentStreamContext` is referenced but its definition is not visible in the retrieved code excerpts; if its `sse_writer` field type does not match `*std.net.Server.Connection`, this is a type error.
84. `handleChatEndpoint` and `handleAgentEndpoint`: After `sendSseHeaders` is called, any subsequent errors that use `sendHttpResponse` will send HTTP response headers after SSE headers have already been sent — the client will receive malformed output. 85. `handleChatEndpoint`: The `session_ev` SSE event is sent before `session_lock` is acquired — a race condition exists where another request could modify the session between the session event being sent and the lock being acquired.
86. `handleChatEndpoint`: `req_data.message` is stored into the session with `global_allocator.dupe` but `req_data` is freed at the end of the function via `defer req_data.deinit(allocator)` — however, the duplication uses `global_allocator`, so the stored copy is independent of `req_data`. 87. `handleChatEndpoint`: The user message's `content` is `MessageContent{ .text = try global_allocator.dupe(u8, req_data.message) }` — but `req_data.message` may contain the user's raw input including any image data (parsed separately). 89. 90. `ExaToolResult`: `tool_type: []const u8` field holds a string literal (either `"search"`) — but the struct has no `deinit` and the caller is responsible for freeing `events` and `tool_msg`. 91. `AgentToolResult`: Has no `deinit` method — callers must manually free `result` and `events`, making resource management error-prone.
92. `ExaMultiResult.deinit`: Does not free `results` (the `?std.json.Value`) — if results is non-null and was obtained from `callExaSingle` which returns a `std.json.Value` from a `parseFromSlice` call whose `parsed.value` is returned directly (with `parsed.deinit()` deferred, see bug #27), the value is already a dangling reference, so not freeing it here is the correct behavior — but this means the entire JSON parsing approach for Exa results is fundamentally broken by use-after-free (#27).
93. `callExaSingle` retry loop: On a network error (`catch |err| { ... continue; }`), the `response_body` ArrayList has `defer response_body.deinit()` — this fires correctly when the block ends on `continue`. 94. `callExaSingle`: On `result.status == .too_many_requests` followed by retry sleep, both `limiter.acquire()` and `exa_limiter.acquire()` are called again at the top of the next loop iteration — this means on retry, the rate limiter is hit again even though the previous request was rate-limited, which is correct behavior but doubles the delay (rate limiter wait + explicit sleep).
95. `writeJsonEvent`: Does not handle `[][]u8` (slice of strings) or other common array types — serializes them as `null`, causing silent data loss for array-valued fields.
96. `getOrCreateSessionLock`: On allocation failure (`try global_allocator.create` or `try global_allocator.dupe`), the error propagates but the partially created mutex (if `create` succeeded but `dupe` failed) is leaked — `lock` is allocated but if `dupe(key)` fails, `lock` is never freed.

98. `generateShortHex`: Returns a slice from `std.fmt.allocPrint` — caller must free it. No bug, but no documentation of ownership.
99. `mimeToExt`: Returns `".img"` as a fallback for unknown MIME types — this is a non-standard extension that most systems will not recognize.
100. `evictSessions`: Does not clean up `session_exa_limiters_map` entries for evicted sessions — unbounded memory growth over time.
